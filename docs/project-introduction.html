Delta Sharing Iceberg 数据共享服务

基于 Delta Sharing 协议，为存储于腾讯云 COS 中的 Apache Iceberg 表格式提供安全、高效的数据共享能力

一、项目背景与目标

在数据驱动的业务场景中，组织内部和跨团队之间经常需要共享数据。 Delta Sharing 是 Databricks 提出的一种开放数据共享协议，定义了数据服务端和客户端之间的标准交互方式。 但官方的 Delta Sharing 实现仅支持 Delta Lake 表格式，无法直接用于 Apache Iceberg 表。

本项目的核心目标是：在完全兼容 Delta Sharing 协议的基础上，扩展支持 Apache Iceberg 表格式， 使存储于腾讯云 COS（对象存储）中的 Iceberg 表可以安全、便捷地共享给内外部数据消费者， 支持使用 Spark、Pandas、Databricks 等主流客户端工具进行数据访问和分析。

核心解决的问题

  • 协议兼容：在 Delta Sharing 协议的框架下，无缝支持 Iceberg 表格式的元数据解析与数据文件导出。
  • 安全共享：通过 Token 认证 + Share 级别授权，精细控制谁能访问哪些表。
  • 直连存储：数据文件通过 COS 预签名 URL 直接从对象存储获取，无需经过服务端中转，避免带宽瓶颈。
  • 客户端兼容：任何支持 Delta Sharing 协议的客户端（Spark、Pandas 等）都可以零修改接入。

二、核心概念

要理解本服务，首先需要了解 Delta Sharing 协议中定义的几个关键概念：

概念 说明
Share 共享单元，一个 Share 包含若干 Schema，是一组数据的逻辑集合。例如一个业务团队可以将自己的所有分析表打包为一个 Share。
Schema 类似于数据库中的 Database 概念，一个 Schema 下包含若干 Table。
Table 数据表，指向 COS 上实际存储的 Iceberg 表。每张表包含其元数据位置、分区信息等。
Recipient 数据接收方（数据消费者）。每个 Recipient 代表一个需要访问数据的用户或系统。
Token 访问凭证（Bearer Token），Recipient 持有 Token 后才能访问被授权的 Share 数据。
Profile 客户端配置文件，包含服务端地址和 Token，客户端加载此文件后即可连接服务获取数据。

四、整体工作流程

阶段一：管理员配置（一次性操作）

1

创建 Share 并配置 Schema / Table

管理员通过 Web 管理后台或 API，定义需要共享的数据集。创建一个 Share（如 "销售数据共享"）， 在其下添加 Schema（对应 DLC 中的数据库），并为每个 Schema 添加具体的 Table（对应 COS 上的 Iceberg 表）。

2

同步 DLC 表（可选）

如果使用了腾讯云 DLC 服务来管理 Iceberg 表，可以通过"同步"功能一键将 DLC 数据库中的所有 Iceberg 表 批量导入到 Share 中，无需手动逐一添加。

3

创建 Recipient 并授权

为数据消费者创建一个 Recipient 身份，然后将其与指定的 Share 关联（授权）， 使该 Recipient 有权访问该 Share 下的所有表。

4

生成 Token 和 Profile

为 Recipient 生成访问 Token，并下载 Profile 配置文件。 Token 明文仅在生成时返回一次，服务端只存储其 SHA-256 哈希值。 Profile 文件下载一次后即从服务端销毁。

5

分发 Profile 给客户端

将 Profile 文件（.share 格式）通过安全渠道交付给数据消费者， 消费者将其配置到自己的客户端工具中即可开始使用。

阶段二：客户端数据访问（日常使用）

1

客户端加载 Profile 并携带 Token 请求

客户端通过 Profile 文件中的服务端地址和 Token，向 Delta Sharing Server 发起请求。 例如：列出可访问的 Share、Schema 和 Table 列表。

2

服务端验证 Token 和权限

服务端收到请求后，对 Token 进行 SHA-256 哈希，与数据库中的哈希值比对。 同时检查 Token 是否过期、是否被撤销、Recipient 是否激活，以及是否对该 Share 有访问权限。

3

解析 Iceberg 表元数据

当客户端查询某张表的数据时，服务端执行以下解析流程：

  1. 从 COS 下载表的 metadata.json（Iceberg 表的根元数据文件），确定当前快照。
  2. 根据快照信息，下载 Manifest List（Avro 格式），获取所有 Manifest 文件列表。
  3. 逐一解析 Manifest 文件（Avro 格式），提取每个数据文件（Parquet）的路径、分区信息、统计信息（min/max/null count）。
  4. 将 Iceberg 的 Schema 转换为 Delta Sharing 协议兼容的 Schema 格式。
4

谓词下推与分区裁剪（性能优化）

如果客户端在查询时传入了过滤条件（如 WHERE ds > '2024-01-01'）， 服务端会在解析元数据的过程中利用分区信息和文件级统计信息（min/max 值）， 提前过滤掉不可能包含目标数据的文件，减少不必要的文件读取和预签名 URL 生成。

5

生成预签名 URL 并返回

服务端为筛选后的每个 Parquet 数据文件生成带有时效限制（默认 6 小时）的 COS 预签名 URL，以 NDJSON 格式返回给客户端。

6

客户端直接从 COS 下载数据

客户端获得预签名 URL 后，直接连接腾讯云 COS 下载 Parquet 文件并解析数据。 数据文件不经过 Delta Sharing Server 中转，避免了服务端带宽瓶颈。 这就好比服务端只是给了客户端一把"临时钥匙"，客户端拿着钥匙自己去仓库取货。

为什么数据不经过服务端中转？
服务端仅负责元数据解析、权限验证和生成临时下载链接，实际的数据（Parquet 文件） 由 COS 对象存储通过预签名 URL 直接提供给客户端。这种设计使得服务端保持轻量， 适合大规模数据共享场景，数据吞吐量由 COS 的带宽保障，不受服务端限制。

五、Iceberg 元数据解析原理

这是本项目的核心工作——将 Iceberg 表的三层元数据结构转换为 Delta Sharing 协议支持的数据文件列表。 Iceberg 表的元数据分为三层，服务端按层级逐步解析：

Iceberg 表的三层元数据

层级 文件 格式 内容
第一层 metadata.json JSON 表的 Schema 定义、分区规则、快照列表（指向 Manifest List 的路径）
第二层 Manifest List Avro 列出所有 Manifest 文件的路径、每个 Manifest 中数据的分区范围
第三层 Manifest Avro 列出实际数据文件（Parquet）的路径、每条记录的统计信息（min/max/null count）

六、核心功能特性

Iceberg 表格式支持

完整解析 Iceberg 三层元数据（metadata → manifest-list → manifest），支持所有 Iceberg 字段类型，包括嵌套的 struct、list、map 等复杂类型。

腾讯云 COS 直连

数据文件通过 COS 预签名 URL 直接下载，不经过服务端中转。预签名 URL 支持自定义有效期（默认 6 小时）。

DLC 自动同步

可选集成腾讯云 DLC（数据湖计算）服务，一键将 DLC 中管理的 Iceberg 表批量同步到 Share 中，支持全量和增量模式。

细粒度访问控制

基于 Recipient-Share 的授权模型，每个 Recipient 只能访问被授权的 Share，未授权访问返回 403。

谓词下推 & 分区裁剪

利用 Iceberg manifest 中的文件级统计信息（min/max/null count），在服务端提前过滤不相关数据文件，大幅减少客户端需要处理的数据量。

时间旅行查询

支持按版本号或时间戳查询历史快照数据，方便数据回溯和问题排查。

安全 Token 管理

Token 明文仅生成时返回一次，服务端仅存储 SHA-256 哈希。支持 Token 过期、撤销、轮换。Profile 文件一次下载即销毁。

审计日志

所有 API 请求自动记录审计日志（JSONL 格式，按日轮转），包含请求来源、操作类型、访问资源、错误码等，满足合规审计需求。

Web 管理后台

提供直观的图形化管理界面，用于创建和管理 Share、Schema、Table、Recipient、Token，以及查看审计日志。

多客户端兼容

与 Delta Sharing 协议完全兼容，支持 PySpark、Pandas、Databricks 等客户端无缝接入，无需客户端做任何改造。

七、安全机制

认证与授权流程

1

客户端携带 Bearer Token 发起请求

所有 Data Plane API 请求必须包含 Authorization: Bearer <token> 请求头。

2

SHA-256 哈希比对

服务端对收到的 Token 进行 SHA-256 哈希，与数据库中存储的哈希值比对。 Token 明文永不落库，仅哈希值被持久化存储。

3

四重状态检查

验证 Token 是否存在、是否已被撤销、是否已过期、关联的 Recipient 是否处于激活状态。

4

Share 级别授权检查

验证 Recipient 是否被授予了对请求中指定 Share 的访问权限。未授权访问返回 403。

其他安全措施

  • 404 安全模糊化：不存在的 Token 与已过期的 Token 返回相同的错误信息，防止信息泄露。
  • Token 配额控制：每个 Recipient 最多持有 2 个有效 Token（可配置），防止凭证滥用。
  • Profile 一次性下载：Profile 文件含敏感凭证信息，下载后即从服务端删除。
  • 预签名 URL 时效控制：COS 预签名 URL 具有时间限制（默认 6 小时），过期后无法访问。
  • 密钥分离：云服务密钥通过环境变量注入，不硬编码在配置文件中。

八、部署与端口说明

服务端启动后监听两个端口，各司其职：

端口 用途 访问范围 认证方式
8088 Data Plane — 数据共享协议接口 对外公开（或按需限定） Bearer Token 认证
8089 Admin API — 管理接口 仅 localhost（确保管理操作安全） 无需额外认证（仅本地可访问）
生产环境部署建议：
建议在前端挂载 Nginx 或其他反向代理，启用 HTTPS，并配置访问控制策略。 8089 端口的 Admin API 仅监听 127.0.0.1，可通过 VPN + SSH 隧道等方式安全访问管理后台。

九、支持的客户端类型

客户端类型 使用方式 适用场景
PySpark 通过 delta-sharing-spark 连接器，以 DataFrame 方式读取共享表数据 大规模数据分析、ETL 处理
Pandas 通过 delta-sharing Python SDK 的 load_as_pandas() 方法 小规模数据探索、快速预览
Databricks 在 Databricks 工作区中通过 Delta Sharing Connector 直接接入 云端数据协作
自定义客户端 任何遵循 Delta Sharing 协议的 HTTP 客户端，发送 REST API 请求即可 自定义集成场景

无论使用哪种客户端，接入流程一致：加载 Profile 文件 → 列出可访问的资源 → 查询具体表的数据。 客户端无需关心底层是 Iceberg 表还是 Delta Lake 表，因为服务端在协议层面做了透明转换。

十、总结

Delta Sharing Iceberg 数据共享服务是一个轻量、安全、高效的数据共享中间层， 它桥接了 Iceberg 表格式与 Delta Sharing 协议之间的差异，使得任何遵循 Delta Sharing 协议的客户端 都可以无缝访问存储于腾讯云 COS 中的 Iceberg 表数据。

一句话概括核心职责：
管理员配置"谁能访问哪些数据"，服务端负责验证身份和权限、解析 Iceberg 元数据、生成临时下载链接， 客户端拿着链接直接从对象存储下载数据文件。
适用场景：
跨团队数据共享、数据湖数据对外提供、BI 分析工具数据接入、机器学习训练数据供给等。
项目核心价值
安全 Token + Share 两级权限控制，预签名 URL 限时访问，审计日志全程追踪。
高效 数据直传不中转，谓词下推减少无效数据，请求级缓存避免重复下载。
开放 遵循开放协议，不锁定客户端，不锁定存储，与现有数据生态无缝融合。
